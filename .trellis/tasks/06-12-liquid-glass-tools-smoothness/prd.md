# Liquid Glass Tools Smoothness Verification

## Goal

Continue the site-wide push toward an extremely smooth Liquid Glass tool station: tool pages should stay responsive while users type, paste large text, parse, format, hash, generate, scroll, switch pages, copy, import, and export. The existing Liquid Glass visual language must remain intact; performance work should primarily bound computation, rendering, data structures, and loading behavior rather than simplifying the interface.

This task is an execution pass toward the active thread goal, not a smaller replacement for it. The broader goal remains active until current evidence proves every tool page satisfies the full standard.

## What I Already Know

* The user requires "input不卡, 切页不卡, 滚动不卡, 复杂解析不卡" across all tools.
* Existing WIP already touches many tool pages under `src/features/**/components`.
* Existing WIP added shared helpers such as `src/utils/outputPreview.ts`, `src/utils/textScan.ts`, and `src/components/ui/input-cap-notice.tsx`.
* The previous active task `06-07-tool-taxonomy-social-polish` records a long series of performance protections: deferred parsing, input caps, bounded previews, action-time exports, row caps, mobile overflow hardening, command-palette lazy loading, and Liquid Glass polish.
* The frontend spec requires expensive client-side text work to use `useDeferredValue`, caps, localized warnings, row limits, `yieldToMain()`, or workers where appropriate.
* The project uses Next.js, React 19, TypeScript, pnpm, Tailwind, localized `src/locales/en.json` and `src/locales/cn.json`, and Liquid Glass primitives in global CSS/UI components.

## Requirements

* Preserve the existing Liquid Glass visual system: global glass background, blur, refraction, highlights, motion language, rounded clipping, hover treatment, and polished mobile layout.
* Verify and repair large text inputs so expensive parsing, formatting, hashing, generation, regex, diffing, previewing, and audit work is deferred, capped, explicitly triggered, or rendered through bounded previews.
* Keep long lists and audit/parse result views bounded in render cost; full copy/download/export payloads may be generated on explicit user action.
* Keep high-frequency time updates shared or isolated so route cards do not each create independent one-second render loops.
* Keep route-only heavy dependencies lazy or isolated to the page/user action that needs them.
* Ensure new and old tool pages consistently have localization, menu discoverability, page metadata, copy/download/export affordances where relevant, input protection, empty states or useful audit feedback, and mobile-safe layout.
* Fix concrete issues found by lint, typecheck, route smoke tests, browser console checks, untranslated-key scans, and mobile overflow checks.
* Avoid overwriting unrelated WIP changes; work with existing edits in place.

## Acceptance Criteria

* [ ] `pnpm run lint` passes.
* [ ] `pnpm run typecheck` passes.
* [ ] Key tool routes return HTTP 200 in both practical locales where smoke-tested.
* [ ] Mobile smoke checks show no document-level horizontal overflow caused by tool content.
* [ ] Browser console smoke checks show no obvious runtime errors on sampled heavy routes.
* [ ] Sampled Chinese and English pages do not expose raw translation keys in visible UI.
* [ ] Large-input sampled tools cap state, computation, rendered rows, or output previews and show localized feedback when truncating or previewing partial data.
* [ ] Liquid Glass visual primitives are not removed or replaced with flat fallback UI during performance fixes.
* [ ] Any newly discovered repeated performance pattern is shared through existing helpers or a narrowly scoped new helper when reuse is clearly justified.

## Definition of Done

* Static checks are green or any remaining failures are explicitly tied to pre-existing/unrelated work with evidence.
* Browser/runtime smoke evidence covers representative converter, formatter, generator, encryption/hash, preview, and social pages.
* Remaining full-site verification gaps are recorded clearly instead of being presented as complete proof.
* The active thread goal is marked complete only after current evidence proves every explicit requirement in the full user objective.

## Technical Approach

1. Establish evidence first: inspect WIP, run lint and typecheck, and identify concrete failures.
2. Fix failures in the smallest compatible way, following existing Liquid Glass UI and helper patterns.
3. Prefer shared helpers already present in the WIP (`outputPreview`, `textScan`, input-cap notice) instead of duplicating new cap/preview logic.
4. Use browser smoke tests on representative high-risk routes with large pasted content and mobile viewport checks.
5. Keep a requirement-by-requirement completion audit before any claim that the long-running goal is finished.

## Decision (ADR-lite)

**Context**: The objective spans many pages and existing WIP already contains broad performance changes. Starting by rewriting individual pages would risk regressions and visual drift.

**Decision**: Use an evidence-driven verification-and-repair pass. Run project gates, inspect current WIP patterns, fix only concrete remaining failures, then smoke-test representative routes before deciding whether broader verification is still missing.

**Consequences**: This keeps the Liquid Glass system stable and avoids churn, but it means goal completion requires stronger evidence than one pass can assume. If the evidence is incomplete, the goal remains active with a clear next target.

## Out of Scope

* Replacing the Liquid Glass design system with a flatter or simpler visual system.
* Moving or renaming existing route URLs unless a separate product decision requires it.
* Adding external services for policy, finance, or web-performance data.
* Claiming every page is complete from narrow smoke tests alone.
